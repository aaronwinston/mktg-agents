# Developer fluency rubric

## Score definitions

### Score 5 (Excellent)
A senior developer would read this and think: 'This person understands the problem.' The content describes real workflows, names actual failure modes, and respects the reader's intelligence.

### Score 3 (Acceptable)
The content is mostly developer-fluent but has some vague benefit language or missing examples. A developer would find it useful but not exceptional.

### Score 1 (Poor)
The content is not developer-fluent. It uses vague benefit language, ignores failure modes, treats agents as magic, or does not include concrete examples.

## Reviewer questions

- Would a developer trust this?
- Does it describe a real workflow or an abstract capability?
- Are failure modes named?
- Are examples concrete and realistic?
- Does it respect the reader's existing technical knowledge?

## Common failure modes

- Too much category language
- No concrete examples or workflow steps
- Treating agents as magic instead of systems with failure modes
- Writing for executives instead of builders
- Vague benefit claims instead of specific product behavior
