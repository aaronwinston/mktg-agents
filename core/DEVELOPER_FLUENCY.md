# Developer Fluency

Developer fluency means writing in a way that technical readers recognize as useful, credible, and grounded in real workflows.

## Principles

1. Describe actual workflows, not abstract capabilities.
2. Name real system behaviors.
3. Be precise about failure modes.
4. Avoid vague benefit language.
5. Use examples.
6. Respect the reader's intelligence.
7. Do not over-explain common concepts to technical readers.
8. Explain tradeoffs honestly.
9. Show how someone would actually use the idea.
10. Connect product capabilities to technical jobs to be done.

## Developer-fluent phrasing

Prefer:
- inspect failed traces
- compare outputs across runs
- define eval criteria
- catch regressions before they reach users
- debug tool-calling failures
- measure response quality
- monitor latency and cost per run
- review spans for each agent step
- set a threshold and get alerted when quality drops

Avoid:
- gain deeper insights
- drive transformation
- unlock intelligent workflows
- build better experiences
- achieve AI excellence
- supercharge your pipeline

## Good developer content includes

- Clear problem definition
- Specific examples
- Workflow steps
- Failure modes and edge cases
- Technical constraints and tradeoffs
- Practical recommendations
- Code or pseudo-code when useful
- Links to docs or source material

## Developer content failure modes

- Too much category language, too little product behavior
- Too much executive framing, not enough builder perspective
- No concrete examples
- Claims without evidence
- Confusing observability with evaluation
- Confusing monitoring with debugging
- Treating agents as magic instead of systems with failure modes
- Writing around the topic instead of explaining it
