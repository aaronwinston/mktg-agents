# 2. Operating philosophy

Build the system around these principles:

## 2.1 Useful before promotional

Every asset should help a technical reader understand, evaluate, build, debug, deploy, or improve something.

## 2.2 Specificity beats hype

Never say “unlock seamless AI workflows” when you can say “trace every tool call, inspect failed steps, and compare eval results across versions.”

## 2.3 Claims require evidence

The system must distinguish between:

- Verified claims
- Reasonable claims
- Claims that require source review
- Claims that should be removed
- Claims that need customer, product, legal, or analyst approval

## 2.4 Developer marketing is enablement

Developer-facing marketing should teach, clarify, compare, demonstrate, or help someone make a better technical decision.

## 2.5 Agents should have editorial separation of duties

Do not make one “content agent” responsible for everything.

Separate:

- Strategy
- Research
- Drafting
- Developer review
- Technical fact-checking
- SEO
- Social distribution
- Copy editing
- Claims risk
- Final publish review
- Content operations

## 2.6 Strong workflows beat strong prompts

The highest-quality output comes from chaining specialists in the right order.

Example:

```txt
brief
→ editorial-director
→ ai-researcher
→ dev-copywriter
→ dev-reviewer
→ technical-fact-checker
→ seo-strategist
→ copy-chief
→ final-publish-reviewer
→ social-editor
→ content-ops-manager
```

## 5.1 `core/VOICE.md`

```md
# Voice

This repository is designed for developer-first content and communications.

The voice should feel:

- Technical but not academic
- Clear but not simplistic
- Confident but not inflated
- Practical but not dry
- Strategic but grounded
- Category-aware but not buzzword-heavy

## Core voice rules

1. Be useful before being persuasive.
2. Prefer concrete product behavior over abstract claims.
3. Explain why something matters to builders.
4. Avoid hype.
5. Avoid vague transformation language.
6. Avoid em dashes.
7. Use short, direct sentences.
8. Use clear section headings.
9. Write like someone who understands the product.
10. Do not over-explain obvious concepts to technical readers.

## Words and phrases to avoid

Avoid these unless there is a specific reason:

- unlock
- unleash
- seamless
- revolutionary
- game-changing
- cutting-edge
- robust
- powerful
- next-generation
- end-to-end
- supercharge
- mission-critical, unless literally true
- AI-powered, unless necessary
- transform the way teams work

## Preferred language

Prefer:

- evaluate
- trace
- debug
- observe
- measure
- compare
- ship
- monitor
- improve
- verify
- inspect
- production
- feedback loop
- failure mode
- workflow
- system behavior
- signal
- evidence
- regression
- dataset
- test case
- span
- trace
- prompt
- output
- evaluator
- experiment

## Good writing sounds like

"An agent is only useful if you can tell whether it worked. Evals give teams a repeatable way to measure agent behavior, catch regressions, and improve performance over time."

## Weak writing sounds like

"Our revolutionary AI platform unlocks seamless agentic workflows that transform how teams build next-generation applications."

## Voice checklist

Before finalizing copy, ask:

- Is this useful to a technical reader?
- Does it say what the product actually does?
- Are the claims specific?
- Is there unnecessary hype?
- Could a developer trust this?
- Would this sound credible coming from a technical founder?
```

## 5.2 `core/STYLE_GUIDE.md`

```md
# Style Guide

## Sentence style

Use short, direct sentences.

Prefer:

"Teams need to know why an agent failed."

Avoid:

"In today’s fast-moving AI landscape, teams are increasingly looking for ways to better understand the underlying reasons why their autonomous systems may not be performing as intended."

## Paragraphs

Keep paragraphs short.

One idea per paragraph.

Use bullets when they improve clarity.

## Headings

Use headings that tell the reader what they will learn.

Prefer:

- "How evals catch regressions"
- "Why tracing matters for agents"
- "What to measure before production"

Avoid:

- "Introduction"
- "Unlocking value"
- "A better way forward"

## Technical examples

Use examples when possible.

Examples can include:

- Agent steps
- Tool calls
- Evaluation criteria
- Failure modes
- Before and after outputs
- Trace inspection
- Regression analysis
- Prompt or system instruction changes

## Avoid em dashes

Use commas, periods, semicolons, or parentheses instead.

## Avoid filler openings

Avoid:

- In today’s world
- As AI continues to evolve
- Now more than ever
- It goes without saying
- There is no doubt that

## Preferred structure for technical blogs

1. Problem
2. Why it matters
3. Technical concept
4. Workflow or example
5. How to evaluate
6. Common failure modes
7. Practical takeaway

## Preferred structure for launch copy

1. What changed
2. Who it helps
3. Why it matters
4. How it works
5. How to get started
6. Proof or context
7. CTA

## Preferred structure for social posts

1. Hook
2. Specific observation
3. Why it matters
4. Practical detail
5. CTA or question
```

```md
# Brand Voice

## Voice attributes

The brand voice is:

- Builder-oriented
- Evidence-led
- Technically credible
- Direct
- Useful
- Curious
- Category-aware
- Pragmatic

## The brand believes

- AI systems need to be evaluated, observed, and improved continuously.
- Agents are software systems with new failure modes.
- Teams need feedback loops, not just dashboards.
- Evals are how teams turn subjective quality into measurable system behavior.
- Observability helps teams understand what happened.
- Evaluation helps teams decide whether it was good.
- The best AI teams improve systems with evidence.

## Sounds like

"Shipping an agent is not the finish line. Once it starts interacting with users, tools, and changing context, teams need a way to measure whether it is still behaving as intended."

## Does not sound like

"We empower enterprises to unlock the full potential of AI transformation through next-generation agentic innovation."

## Tone by channel

### Blog

Educational, practical, technical.

### Launch

Clear, specific, restrained, outcome-oriented.

### Social

Sharper and more opinionated, but still credible.

### Analyst relations

Strategic, precise, proof-oriented.

### Customer stories

Concrete, evidence-led, customer-centered.

### Executive comms

Opinionated, technical, and grounded in category insight.
```

## 5.6 `core/CLAIMS_POLICY.md`

```md
# Claims Policy

Every factual claim must be reviewed before publication.

## Claim categories

### Verified

The claim is supported by approved source material.

Example:

"Phoenix supports tracing LLM application calls."

### Needs source

The claim may be true, but no source is provided.

Example:

"Most teams struggle to evaluate agents."

### Needs product review

The claim describes product behavior, feature availability, roadmap, or technical implementation.

Example:

"The platform automatically detects regressions across agent runs."

### Needs legal review

The claim touches compliance, guarantees, regulated industries, security, privacy, or contractual obligations.

Example:

"Built for HIPAA-compliant AI workflows."

### Needs customer approval

The claim references a customer name, result, deployment, metric, quote, or use case.

Example:

"Customer X reduced debugging time by 40%."

### Needs analyst/source citation

The claim references third-party research, analyst commentary, rankings, or market data.

Example:

"Analysts expect agent observability to become a major category."

### Remove or soften

The claim is too broad, too risky, or not supportable.

Example:

"Our platform guarantees agent reliability."

## Safer language patterns

Instead of:

"Guarantees reliable agents."

Use:

"Helps teams evaluate agent behavior and catch regressions before they reach users."

Instead of:

"Eliminates hallucinations."

Use:

"Helps teams detect unsupported or incorrect outputs through evaluation workflows."

Instead of:

"Fully automates AI governance."

Use:

"Supports governance workflows by making traces, evaluations, and system behavior easier to inspect."

## Required output from claims review

Every claims review should include:

- Verified claims
- Claims needing source
- Claims needing product review
- Claims needing legal review
- Claims needing customer approval
- Claims to remove or soften
- Suggested safer language
```

```md
## The editorial promise

Every asset should make the reader smarter.

## Principles

### 1. Start with the reader's problem

Great content should help the market understand what matters.

Credibility compounds.
```

## 15.2 `scripts/lint_skill_files.py`

```python
from pathlib import Path
import sys

REQUIRED_SECTIONS = [
    "---",
    "# Role",
    "# Use this skill when",
    "# Do not use this skill when",
    "# Inputs expected",
    "# Source hierarchy",
    "# Process",
    "# Output format",
    "# Quality bar",
    "# Failure modes to avoid",
    "# Related skills",
]

def lint_skill(path: Path) -> list[str]:
    text = path.read_text(encoding="utf-8")
    errors = []

    for section in REQUIRED_SECTIONS:
        if section not in text:
            errors.append(f"{path}: missing section `{section}`")

    if "description:" not in text:
        errors.append(f"{path}: missing frontmatter description")

    if "name:" not in text:
        errors.append(f"{path}: missing frontmatter name")

    return errors

def main() -> int:
    skill_files = list(Path("skills").rglob("SKILL.md"))
    all_errors = []

    if not skill_files:
        print("No SKILL.md files found.")
        return 1

    for path in skill_files:
        all_errors.extend(lint_skill(path))

    if all_errors:
        print("Skill lint failed:")
        for error in all_errors:
            print(f"- {error}")
        return 1

    print(f"Skill lint passed for {len(skill_files)} skills.")
    return 0

if __name__ == "__main__":
    sys.exit(main())
```

## 15.3 `scripts/generate_skill_index.py`

```python
from pathlib import Path

def extract_frontmatter(text: str) -> dict:
    if not text.startswith("---"):
        return {}

    end = text.find("---", 3)
    if end == -1:
        return {}

    frontmatter = text[3:end].strip()
    data = {}

    for line in frontmatter.splitlines():
        if ":" in line:
            key, value = line.split(":", 1)
            data[key.strip()] = value.strip()

    return data

def main() -> None:
    skill_files = sorted(Path("skills").rglob("SKILL.md"))
    lines = [
        "# Skills",
        "",
        "This file is generated by `scripts/generate_skill_index.py`.",
        "",
        "## Skill index",
        "",
        "| Skill | Path | Description |",
        "|---|---|---|",
    ]

    for path in skill_files:
        text = path.read_text(encoding="utf-8")
        fm = extract_frontmatter(text)
        name = fm.get("name", path.parent.name)
        description = fm.get("description", "")
        lines.append(f"| {name} | `{path}` | {description} |")

    Path("skills/README.md").write_text("\n".join(lines) + "\n", encoding="utf-8")
    print(f"Wrote skills/README.md with {len(skill_files)} skills.")

if __name__ == "__main__":
    main()
```

## 15.4 `scripts/create_content_brief.py`

```python
from pathlib import Path
import argparse
import shutil
import datetime

TEMPLATES = {
    "blog": "briefs/blog-brief-template.md",
    "launch": "briefs/launch-brief-template.md",
    "case-study": "briefs/case-study-brief-template.md",
    "analyst": "briefs/analyst-comms-brief-template.md",
    "founder-social": "briefs/founder-social-brief-template.md",
    "technical-guide": "briefs/technical-guide-brief-template.md",
    "newsletter": "briefs/newsletter-brief-template.md",
    "campaign": "briefs/campaign-brief-template.md",
}

def slugify(value: str) -> str:
    return "".join(c.lower() if c.isalnum() else "-" for c in value).strip("-").replace("--", "-")

def main() -> None:
    parser = argparse.ArgumentParser(description="Create a content brief from a template.")
    parser.add_argument("type", choices=TEMPLATES.keys())
    parser.add_argument("title")
    parser.add_argument("--out-dir", default="briefs/drafts")
    args = parser.parse_args()

    template = Path(TEMPLATES[args.type])
    if not template.exists():
        raise FileNotFoundError(f"Template not found: {template}")

    out_dir = Path(args.out_dir)
    out_dir.mkdir(parents=True, exist_ok=True)

    date = datetime.date.today().isoformat()
    filename = f"{date}-{slugify(args.title)}.md"
    destination = out_dir / filename

    shutil.copyfile(template, destination)
    print(f"Created {destination}")

if __name__ == "__main__":
    main()
```

## 15.5 `scripts/run_editorial_check.py`

```python
from pathlib import Path
import argparse
import re

RISKY_PHRASES = [
    "unlock",
    "unleash",
    "revolutionary",
    "game-changing",
    "seamless",
    "next-generation",
    "guarantee",
    "eliminate hallucinations",
    "fully automated",
]

def main() -> None:
    parser = argparse.ArgumentParser(description="Run a basic editorial check on a markdown file.")
    parser.add_argument("file")
    args = parser.parse_args()

    path = Path(args.file)
    text = path.read_text(encoding="utf-8")
    lower = text.lower()

    print(f"Editorial check: {path}")
    print()

    found = []
    for phrase in RISKY_PHRASES:
        if phrase in lower:
            found.append(phrase)

    if found:
        print("Potential risky or weak phrases:")
        for phrase in found:
            print(f"- {phrase}")
    else:
        print("No risky phrases from the basic list were found.")

    em_dash_count = text.count("—")
    if em_dash_count:
        print(f"\nFound {em_dash_count} em dash characters. Consider replacing them.")

    headings = re.findall(r"^#+\s+", text, flags=re.MULTILINE)
    print(f"\nHeading count: {len(headings)}")

if __name__ == "__main__":
    main()
```
