# Developer Marketing Workflow Principles

## Purpose

Use this context layer whenever ForgeOS is creating or reviewing developer-facing campaigns, landing pages, ads, product pages, launch copy, or technical enablement content.

The core principle:

> Developer marketing works best when it doubles as developer enablement.

A developer-facing asset should teach the reader something concrete about how the product helps them do real work.

---

## Core doctrine

### 1. Show the workflow, not the feature

Do not lead with abstract capabilities.

Prefer:

- Plan the change
- Generate the diff
- Review the PR
- Merge when checks pass

Avoid:

- Powerful AI
- Transform developer productivity
- Supercharge your workflow
- Agentic innovation

If the claim cannot be represented as a concrete workflow, it is probably not ready for developer marketing.

---

### 2. Use product truth as the creative

Developers trust:

- commands
- terminal output
- diffs
- test results
- pull requests
- CI checks
- issue references
- real file names

They distrust:

- abstract AI visuals
- generic productivity claims
- polish that hides product behavior
- copy that could apply to any tool

When possible, let the product interface carry the message.

---

### 3. Pre-qualify aggressively

Developer-facing assets do not need to be legible to everyone.

They need to be immediately legible to the right developer.

It is acceptable if an ad filters out non-technical viewers. In developer marketing, specificity can increase conversion quality.

---

### 4. The PR is the accountability boundary

For agentic developer workflows, the most credible outcome is not “AI wrote code.”

The credible outcome is:

> A reviewable PR with a diff, checks, and human review.

Whenever possible, end workflow narratives in a pull request, a passing check, or a reviewable diff.

---

### 5. Teach while selling

Every asset should leave the reader with a clearer understanding of:

- what command to run
- what workflow to try
- what output to expect
- what they should inspect before merging

If the asset only persuades but does not teach, it is underperforming.

---

## Durable campaign patterns

### Pattern: Problem → command → artifact

Use when explaining CLI, agent, or automation features.

Structure:

1. Start with a recognizable developer problem
2. Show the command or workflow
3. End with a concrete artifact

Examples:

- Missing tests → generate coverage plan → PR
- Stack trace → root-cause analysis → passing tests
- TODO → delegated task → PR
- Migration plan → repo changes → reviewable diff

---

### Pattern: Plan → execute → review → ship

Use for multi-step agent workflows.

Structure:

1. Plan the change
2. Execute across files or models
3. Review the diff
4. Ship through GitHub

This pattern is especially useful when the product might feel risky or autonomous.

---

### Pattern: Interface as proof

Use when trust is the barrier.

Instead of explaining the feature in prose, show:

- terminal prompt
- generated plan
- files changed
- CI checks
- PR title
- merge button

The viewer should be able to infer the value from the UI.

---

## Copy rules

### Prefer

- Turn TODOs into PRs
- Find missing tests
- Review one diff
- Merge when checks pass
- Assign the task
- Ship the PR
- Diagnose the failure
- Fix the root cause

### Avoid

- Unlock productivity
- Supercharge development
- Revolutionary AI
- Seamless magic
- Fully autonomous
- Unlimited
- Effortless

---

## Quality bar

A strong developer campaign asset should be:

- specific
- workflow-led
- low-hype
- product-true
- reviewable
- technically credible
- easy to reproduce
